# Reference

在 interview 已经确定“已知的未知”后，锚定能被证明的事实，再进入 planning。把每条“已知的已知”绑定到可追溯参照物，避免把记忆、猜测或过时文档当成事实。

## 前置条件与范围

1. 确认已有 interview 产出的访谈结论；若没有，先执行 interview。
2. 从访谈结论提取最终需求、验收标准、关键约束和采用的默认值。
3. 用一句话写明当前要确认的对象与边界。
4. 列出需要参照物回答的事实问题，例如：
   - 当前系统实际怎样工作？
   - 哪些接口、格式和约束已被固定？
   - 已有相似实现遵循什么模式？
   - 什么行为已被测试或文档承诺？
5. 不在本流程中设计新方案；这里只建立 planning 所需的事实基线。

## 搜集参照物

优先使用专用读取和搜索工具。按任务相关性检查：

- 需求、设计文档、ADR、README、API 文档
- 实现代码、公开接口、类型定义、数据库 Schema
- 测试、fixture、快照、示例输入输出
- 配置、部署清单、迁移脚本、生成文件
- git 历史、issue、PR、变更记录
- 用户提供的样例、竞品或目标效果
- 官方规范、版本化外部文档

只搜到足够回答事实问题为止，避免无边界地罗列资料。

## 判断来源权威性

根据事实类型选择权威来源，而不是机械采用统一优先级：

- **当前运行行为**：可复现测试或运行证据 > 当前代码 > 文档
- **对外承诺**：已发布规范、API 契约或需求文档 > 当前实现
- **设计意图**：ADR、评审记录、相关 commit/PR > 代码形态猜测
- **用户目标**：用户明确确认 > 样例和既有行为

记录版本、分支、提交或访问日期。来源过时、生成方式不明或与当前版本不匹配时，降低可信度。

## 提取已知事实

每条事实必须满足：

1. 用陈述句表达，不夹带建议。
2. 附可定位证据：文件路径与行号、测试名称、commit、issue/PR 或 URL 与版本。
3. 标注证据强度：
   - **已验证**：由测试、运行结果或多个一致来源直接证明
   - **已记载**：权威文档明确声明，但尚未运行验证
   - **仅观察**：当前代码如此，但无契约或测试保证
4. 明确适用范围，例如平台、版本、环境或调用路径。

没有证据的内容不得归入“已知的已知”；标记为 planning 必须显式处理的假设或风险。若证据直接推翻访谈中的关键需求前提，返回 interview 让用户裁决。

## 处理冲突

发现文档、代码、测试或历史互相矛盾时：

- 并列呈现冲突来源，不擅自选边或融合
- 说明冲突会影响哪个决策
- 能用只读验证消解时立即验证
- 不能消解时标记为“待裁决”，不归入事实基线

## 输出参考基线

```markdown
## 参考基线：<对象>

### 权威参照物
- <来源> — <它能证明什么> — <版本/范围>

### 已知的已知
- [已验证|已记载|仅观察] <事实>（证据：<精确出处>；范围：<适用条件>）

### 冲突与待裁决
- <来源 A> 与 <来源 B> 的冲突 → <影响>

### 非事实项
- <常见假设或推断> → 因 <缺少何种证据> 未纳入基线

### Planning 输入
- 已固定事实：<planning 可以直接采用的约束>
- 待验证假设：<planning 需安排低成本验证的事项>
- 待裁决冲突：<若存在，先返回 interview>
```

将参考基线和访谈结论一起交给 planning。后续计划与实现都引用这份基线；新证据推翻旧事实时，先更新基线再继续。
