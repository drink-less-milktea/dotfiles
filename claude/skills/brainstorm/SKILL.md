# Brainstorm（头脑风暴与原型）

盲区扫描回答"我们不知道什么"；本技能回答"用户知道但没说出来的是什么"。通过发散方案、结构化提问和最小原型，把用户脑中的隐性偏好、约束、验收标准逼出来，变成显性需求。

## 前置条件（硬性）

必须已有一份 blindspot-pass 产出的盲区报告。若没有：

- 先执行 `~/.claude/skills/blindspot-pass/SKILL.md` 的流程生成报告，再回到本技能
- 不要跳过扫描直接头脑风暴——没有事实基础的发散只是猜测

## 工作流

### 第 1 步：从盲区报告提取分叉点

从报告中列出所有会导致方案分岔的项：🔴/🟡 发现、未回答的澄清问题、有多种解读的需求点。每个分叉点都是一个待显性化的"未知的已知"——用户心里多半有答案，只是没说。

### 第 2 步：发散候选方案

围绕分叉点生成 3–5 个**实质不同**的方案方向（不是同一方案的三档配置）。每个方向用固定格式描述：

```markdown
### 方向 N：<一句话概念>

- 核心取舍：牺牲什么换什么
- 适合场景：什么情况下这是对的选择
- 检验的隐性假设：选它意味着用户其实在乎 <X> 而非 <Y>
```

方向之间必须在关键分叉点上给出不同答案，让用户的选择本身携带信息。

### 第 3 步：结构化提问

用 AskQuestion 工具提问（不可用时在回复中列出问题）。规则：

- 每轮最多 3 问，每问的选项直接对应第 2 步的方向或分叉点
- 只问选择后会实质改变做法的问题；能从代码库或上下文推断的不问
- 每个选项标注一句后果（"选 A 意味着放弃 B 的 <某能力>"）
- 用户的每个回答都记下来——那就是刚显性化的一条"未知的已知"

### 第 4 步：最小原型

对收敛出的 1–2 个方向做可丢弃原型，目的是引出反应而非交付代码：

| 需求类型    | 原型形式                            |
| ----------- | ----------------------------------- |
| 交互/界面   | 可运行的界面骨架或静态 mock         |
| 数据流/架构 | mermaid 图 + 关键接口签名           |
| 算法/逻辑   | 20–50 行核心逻辑草稿 + 样例输入输出 |
| 文档/内容   | 一页样张                            |

原型规则：

- 明确标注"原型，随时可丢弃"，放在独立目录或临时文件，不混入正式代码
- 只做到能让用户说"对，就是这个"或"不对，我要的是……"的最低程度
- 展示原型后必须问一个开放问题："哪里和你想的不一样？"——差异处就是残留的隐性需求

### 第 5 步：收敛为显性需求清单

汇总产出，作为后续 interview 的输入：

```markdown
## 需求确认单：<需求一句话>

### 已显性化的决策（原"未知的已知"）

- <决策> ← 来源：<第几轮提问/原型反馈>

### 选定方向

<方向 N 及理由>

### 验收标准

- <用户认可的可检验条件>

### 遗留开放项

- <仍未确认、按默认值处理的点及其默认值>
```

## 注意事项

- 发散与提问合计不超过两轮，防止无限澄清；第二轮后带默认值收敛
- 方案要基于盲区报告中的事实，引用报告条目，不重复调研
- 原型代码不做错误处理、不写测试、不追求风格，快和可丢弃优先
- 收敛后主动交棒给 interview，并附上需求确认单；不要跳过 interview 或 reference 直接进入 planning/实现
