# Explaination

把面向 agent 的计划与执行记录翻译为面向人的交付讲解。不要照搬日志；围绕读者关心的“为什么做、产出了什么、怎么工作、如何验证、还缺什么”组织内容。

## 收集证据

先读取并交叉核对：

1. planning 产出的已批准计划
2. `.claude/notes/*-implementation.md` 执行笔记
3. 最终 git diff 或变更文件
4. 新增和更新的测试及其结果
5. 相关 README、设计文档、API 契约和使用示例

计划说明意图，执行笔记说明过程，代码和测试说明最终事实。三者不一致时，以实际代码与可复现验证为准，并在文档中明确偏差。

缺少计划或执行笔记时不要猜测；列出缺失输入，并只记录能从现有证据确认的事实。

## 确定读者与保存位置

从上下文判断主要读者：

- 用户或产品人员：强调结果、行为和使用方法
- 开发者：补充架构、接口、关键文件和扩展点
- 运维人员：补充配置、部署、迁移、监控和回滚

沿用项目已有文档约定。没有约定时保存为：

`docs/explanations/<plan-slug>.md`

标题和正文使用项目主要语言；术语首次出现时用一句话解释。

## 写作规则

- 先写可感知的产出，再写实现细节
- 区分“计划目标”“实际完成”“未完成或改变”
- 每项关键修改说明修改前、修改后和影响
- 引用文件路径、测试名称、命令或文档位置，使结论可追溯
- 用短段落、列表和小型示例，避免逐行复述代码
- 只有流程或依赖关系较复杂时使用 Mermaid
- 不写密钥、token、内部提示词、大段 agent 日志或无关调试输出
- 不使用“应该已经”“大概”“可能”等无法验证的表述

## 文档结构

按需删减无内容章节，但保留“产出与修改”“验证”“限制”：

```markdown
# <功能或改动名称>

## 概览
<用 2–4 句话说明问题、结果和主要受益者>

## 产出与修改
### <用户可感知产出>
- 修改前：
- 修改后：
- 影响：
- 相关文件：

## 工作原理
<用易懂语言解释主要流程、组件关系或数据流>

## 关键决策
- <决策>：<原因、取舍和影响>（来源：<计划/执行笔记>）

## 使用方式
<最短可运行步骤、命令、配置或示例>

## 验证
- <测试或检查> → <实际结果>

## 计划与实际差异
- <计划内容> → <实际处理及原因>

## 限制与后续
- <真实存在的限制、风险或未完成事项>

## 参考资料
- <计划、执行笔记、代码、测试或相关文档>
```

## 核对产出

保存前逐项检查：

1. 执行笔记中每个已完成计划项都能在文档中找到对应产出。
2. 最终 diff 中每个重要行为变化都已解释；机械性文件改动可以合并描述。
3. 文档声称通过的测试都有实际执行证据。
4. 未完成、失败或偏离计划的内容没有被包装成已完成。
5. 路径、命令和示例与当前代码一致。

只创建或更新讲解 Markdown；不要顺带修改实现、补测试、提交或推送。文档核对完成后将计划、执行笔记、讲解文档、当前 diff 和测试证据交给 quiz；quiz 全部通过前禁止 commit 或 merge。
