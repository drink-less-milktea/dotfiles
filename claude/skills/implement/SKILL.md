# Implement

Cursor 官方没有名为 Goal Mode 的模式。将“goal mode”解释为：切换到可执行写操作的 Agent Mode，并把已批准计划当作目标合同持续推进，直到计划完成或遇到必须由用户裁决的阻塞。

## 前置条件与模式

1. 确认已有 planning 产出的具体计划，且用户已经批准。
2. 若计划未批准、含 TBD 或仍有未选择的高代价决策，停止并返回 planning。
3. 调用 `SwitchMode` 切换到 `agent`；已经处于 Agent Mode 时不要重复调用。
4. 读取 git 状态和计划涉及的代码，区分既有改动，绝不覆盖用户工作。

## 创建执行笔记

在派出任何 agent 前创建：

`.claude/notes/<plan-slug>-implementation.md`

若项目已有 agent 笔记约定，沿用项目路径。该文件由主 agent 单独维护，子 agent 只返回结构化结果，避免并发写冲突。

初始化模板：

```markdown
# Implementation Notes: <目标>

## 执行合同
- 已批准计划：<路径或摘要>
- 成功标准：
- 明确不做：
- 当前状态：in_progress

## 任务与所有权
- [ ] T1 <任务> — agent: <名称> — owns: <文件/目录> — depends_on: <任务>

## 关键决策
### D1 <决定>
- 时间：
- 触发原因：
- 选择与理由：
- 被放弃的选择：
- 影响范围：
- 是否偏离计划：否/是（对应计划项）

## 关键改动
- <文件>：<行为变化>；原因：<决策/计划项>；验证：<测试>

## 验证记录
- <命令或检查> → <结果>

## 阻塞与后续
- <阻塞、风险、遗留事项>
```

笔记面向后续 agent：保持简洁、可恢复、无聊天叙事；不得写入密钥、token 或大段原始日志。记录每一个关键决策，并确保最终 diff 中每个修改、增加、删除或重命名的文件都在“关键改动”中有对应条目；机械性改动可以合并描述，但不能遗漏文件。

## 拆分多 agent 任务

把计划转换为依赖 DAG，再派 agent：

1. 仅并行执行没有依赖且文件所有权不重叠的任务。
2. 在同一批次并行启动独立 agent，明确每个 agent：
   - 目标与对应计划项
   - 可修改的文件或目录
   - 禁止修改的边界
   - 完成标准和测试命令
   - 返回格式：决策、改动文件、验证结果、偏差、阻塞
3. 至少使用实现与独立验证两个角色。无法安全并行时，顺序运行实现 agent 和 review/test agent，仍保持多 agent 交叉验证。
4. 不让多个 agent 修改同一文件；共享接口先由一个所有者落地，再解锁依赖任务。
5. 主 agent 负责整合，不把最终一致性委托给子 agent。

## 执行循环

对每批任务执行：

1. 派发前在笔记中登记任务、所有权、依赖和预期验证。
2. agent 返回后先检查 diff 与测试证据，再更新任务状态。
3. 立即把每个关键决策和改动文件写入笔记，不等到最后回忆补写。
4. 合并跨任务接口，运行受影响范围的测试和 lint。
5. 根据 DAG 解锁下一批任务。
6. 保持至少一个安全、相关、属于计划范围的下一步时持续推进；不要因为一次测试失败就停止。

## 管理计划偏差

- **实现细节偏差**：不改变外部行为、架构或范围时可继续，但记录原因和影响。
- **重大偏差**：会改变高代价决策、公共契约、数据迁移、范围或验收标准时立即暂停；更新笔记中的阻塞，返回 planning 让用户确认。
- 不做计划之外的相邻重构、清理或功能扩展。
- 不自动 commit 或 push，除非用户明确要求。

## 验证与收尾

1. 由独立 agent 对照计划、执行笔记和最终 diff 做完成度审查。
2. 运行计划要求的单元、集成、端到端测试及 lint；记录命令与结果。
3. 修复本次引入的问题并重新验证。
4. 将笔记状态改为 `completed`，记录最终文件清单、已验证标准和未完成事项。
5. 最终回复只概述结果、验证和真实遗留风险，并引用执行笔记路径。
6. 将计划、执行笔记、最终 diff 和测试证据交给 explaination；不要跳过讲解文档直接进入 quiz、commit 或 merge。
