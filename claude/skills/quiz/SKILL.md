# Quiz

把提交或合并前的最后一步变成可追溯验收门禁。测试用户是否正确理解并接受实际交付结果，不考无关记忆题。

## 前置条件

确认以下资料齐全：

1. planning 的已批准计划
2. implement 的执行笔记
3. explaination 的人类可读讲解文档
4. 当前最终 diff
5. 与当前 diff 对应的自动化测试和验证结果

缺少讲解文档时先执行 explaination。实现、文档或测试结果互相矛盾时锁定门禁，先返回 implement 修正，不用测验掩盖问题。

## 门禁一：自动化验证

在向用户提问前：

1. 确认执行笔记列出的单元、集成、端到端测试和 lint 都针对当前 diff 执行。
2. 重新运行过期、缺失或受后续修改影响的检查。
3. 任一必需检查失败时：
   - 标记 `AUTOMATED_GATE=failed`
   - 不开始或继续用户测验
   - 返回 implement 修复并重新运行全部受影响检查
4. 全部必需检查通过后标记 `AUTOMATED_GATE=passed`。

用户测验不能替代自动化测试、CI 或代码审查。

## 建立测验目标

从讲解文档和实际改动提取 3–7 个可验证目标，至少覆盖：

- 主要产出和用户可感知行为
- 正确使用方式或操作路径
- 关键决策及其影响
- 已知限制、失败行为或回滚方式
- 计划与实际存在的重大差异

不考行号、内部变量名、机械性文件列表或讲解文档没有提供的信息。每个目标准备一个有唯一正确判断的首题和一个语义等价但场景不同的重测题。

## 逐题测试

每轮只测试一个目标：

1. 用 AskQuestion 提出一道题；优先使用场景题，提供 2–4 个互斥选项并允许 Other。
2. 不在题目中泄露答案，不使用文字陷阱。
3. 提问后停止，等待用户回答。
4. 根据代码、测试和讲解文档评分，不按措辞是否完全一致评分。

评分后：

- **通过**：说明该目标已通过并引用简短证据，进入下一个未测试目标。
- **不通过或含糊**：说明错误会导致什么实际问题，引用讲解文档或代码证据，将该目标标记为 failed，并立即用等价新场景重新测试；不要重复原题让用户机械记忆。

持续重测失败目标，直到通过或用户明确停止。不要因部分通过提前解锁门禁。

## 持久化状态

在 implement 执行笔记末尾维护：

```markdown
## Quiz Gate
- 基于 diff：<commit/working tree 标识>
- 自动化门禁：passed/failed
- Q1 <目标>：passed/failed — <证据>
- Q2 <目标>：passed/failed — <证据>
- 用户测验：passed/failed/in_progress
- 提交/合并门禁：locked/unlocked
```

记录目标和结果，不在未通过前写出完整答案库。

## 失效与重测

出现任一情况时立即锁定门禁：

- 代码、配置、迁移、测试或讲解文档在测验后发生变化
- 自动化测试结果不再对应当前 diff
- 用户指出讲解与实际行为不一致
- 计划验收标准发生变化

重新运行受影响的自动化验证，并对所有受影响目标出等价新题。未受影响且证据仍有效的目标可保留通过状态。

## 全部通过

仅当同时满足以下条件时标记：

- `AUTOMATED_GATE=passed`
- 所有用户测验目标均为 `passed`
- 当前 diff 与记录的测验基线一致
- 没有未披露的阻塞或失败验收项

然后输出：

```text
QUIZ_GATE=passed
COMMIT_MERGE_GATE=unlocked
```

“unlocked”只表示允许进入 Git 操作，不自动 commit、push 或 merge。仅当用户明确要求对应操作时才执行；merge 前仍需确认 CI、评审和分支保护要求全部满足。
