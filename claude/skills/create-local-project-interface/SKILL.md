# 为远程项目创建本机接口目录

## 这个 skill 做什么

给一个运行在远程服务器上的项目，在本机创建一个**轻量接口目录**。这个目录只保存元信息和连接指引，**不保存任何代码、数据、模型、结果的副本**。

目的：让任何 agent（或人）打开本机这个目录，就能快速了解「项目在哪、怎么连、环境如何、资源在哪」，然后直接 SSH 到远程干活。

## 何时使用

- 适合：项目主体在远程服务器，本机只是「控制台」，需要让 agent 能快速接入
- 不适合：项目代码就在本机（没必要建接口目录，直接读代码即可）
- 不适合：一次性任务（建接口目录收益低于成本）

## 核心原则

1. **接口而非副本**：本目录只放元信息 `.md` 文件，绝不拷贝代码/数据/模型/结果
2. **动态查询优先**：写明「怎么查」，而不是把当前状态抄死（远程状态会变）
3. **路径用绝对路径**：避免歧义，A800/H100 等不同机器路径差异要显式列出
4. **常见陷阱要写**：agent 容易踩的坑（venv 名字、nvcc bug、量化配置等）必须记录

## 执行流程

### 第 1 步：收集项目信息

和用户确认（或从对话推断）以下信息：

| 信息 | 必需 | 示例 |
|------|------|------|
| 远程服务器 SSH 别名 | ✅ | `a800` / `h100` / `lab` |
| 远程项目根路径 | ✅ | `/home/user/project` |
| 本机接口目录放哪 | ✅ | `~/Documents/code/<project-name>/` |
| 项目是做什么的 | ✅ | 一句话描述研究/工程目标 |
| 有几台服务器 | ⚪️ | 单台 or 多台（多台需逐台说明） |
| 模型/数据位置 | ⚪️ | 如果项目和模型/数据分离 |
| 关键环境配置 | ⚪️ | venv 路径、CUDA、特殊工具 |

如果用户没明说，先 `ssh <别名> "ls / pwd"` 探查，或者问清楚再继续。

### 第 2 步：实地核查远程状态

**不要凭记忆写**，要 SSH 上去实际核查：

```bash
# SSH 配置
grep -A4 "Host <别名>" ~/.ssh/config

# 项目结构
ssh <别名> "ls <项目根>"

# 环境信息
ssh <别名> "<venv>/bin/python --version"
ssh <别名> "nvidia-smi --query-gpu=index,name,memory.total --format=csv,noheader"

# 模型/数据位置
ssh <别名> "ls <项目根>/models <项目根>/data 2>/dev/null"
```

每条信息都从实际命令结果来，不要猜。

### 第 3 步：创建接口目录

```bash
mkdir -p <本机路径>
```

### 第 4 步：生成 7 个 `.md` 文件

根据核查结果，按下面的模板写。**不是所有文件都必须有**——按项目实际情况裁剪：

| 文件 | 内容 | 何时需要 |
|------|------|----------|
| `README.md` | 项目总览 + 文件索引 + 快速接入 | ✅ 必有 |
| `ACCESS.md` | SSH 配置 + 路径对照表 | ✅ 必有 |
| `AGENTS.md` | 给 agent 的操作指引 + 常见陷阱 | ✅ 必有 |
| `ENVIRONMENTS.md` | Python/CUDA/GPU 环境 | 涉及计算/实验时 |
| `MODELS.md` | 模型位置 + 校验状态 | 涉及 ML 模型时 |
| `DATASETS.md` | 数据集位置和结构 | 涉及训练/评测数据时 |
| `RESULTS.md` | 结果目录命名规则 | 实验类项目 |

如果项目不涉及模型/数据/实验，可以省略对应文件，但 README/ACCESS/AGENTS 三个必有。

### 第 5 步：写入本机 memory（可选）

如果用户希望以后能快速接入，在 memory 里记一条指针：

文件：`~/.claude/projects/<本项目hash目录>/memory/<project>-interface.md`
```markdown
---
name: <project>-interface
description: <project> 项目的本机接口目录位置和远程接入方式
metadata:
  type: reference
---

<project> 项目有本机接口目录：<本机路径>

只放元信息，不保存代码/数据/模型副本。所有真实资源在远程：
- <服务器别名> → <远程路径>

新 session 接入时先读接口目录的 AGENTS.md 和 ACCESS.md。
```

并在 `MEMORY.md` 索引加一行。

## 质量检查清单

生成接口目录后，逐条检查：

- [ ] 每条远程路径都来自实际 SSH 核查，不是凭记忆
- [ ] 多服务器（如果有）的差异显式列出
- [ ] venv 路径、Python 版本、关键依赖版本准确
- [ ] 模型/数据位置写的是绝对路径
- [ ] 常见陷阱（用户踩过或容易踩的）已收录到 AGENTS.md
- [ ] README 有文件索引和快速接入步骤
- [ ] **没有**拷贝任何代码、数据、模型、结果文件

## 常见陷阱（skill 自身使用时）

1. **别凭记忆写**：远程状态会变，每条信息都要 SSH 实查
2. **别过度生成**：不涉及模型的项目就别造 MODELS.md，按需裁剪
3. **路径要绝对**：`./models/` 这种相对路径在跨机时是灾难
4. **别拷贝内容**：这个 skill 是建接口，不是同步代码。如果要同步代码，那是另一个 skill
5. **memory 要更新**：如果远程路径变了，接口目录的 `.md` 和 memory 指针都要更新

## 示例

参考已建好的接口目录：
- `/Users/yiwencai/Documents/code/moe-kv-cache-experiments/`（MoE KV Cache 项目，A800+H100 双机）

那个项目是「研究类 + 多服务器 + 模型/数据分离」的典型例子，包含全部 7 个文件。如果你的项目更简单（单机、无模型），可以省略 MODELS/DATASETS/RESULTS 等。
