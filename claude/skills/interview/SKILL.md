# Interview

在盲区扫描暴露风险、头脑风暴显性化偏好之后，通过单问题访谈消除仍会影响执行的“已知的未知”。

## 前置检查

1. 确认上下文中已有 brainstorm 产出的需求确认单或候选方案。
2. 若没有，先执行 `~/.claude/skills/brainstorm/SKILL.md`；不要凭空开始访谈。
3. 从需求确认单的“遗留开放项”、未定分叉和验收缺口建立问题队列。

## 问题筛选

仅保留答案会改变至少一项执行决策的问题：

- 实现范围或不做什么
- 架构、技术路线或数据契约
- 优先级、性能、安全或兼容性取舍
- 失败策略、迁移方式或回滚方式
- 可观察、可验证的验收标准

先从代码、文档或现有上下文查证能自行回答的问题。不要询问偏好性闲话、重复已确认内容，或答案不影响执行的问题。

按“方向改变 > 风险消除 > 依赖解锁 > 细节完善”排序问题。

## 单问题访谈循环

每一轮严格执行：

1. 只提出队列中优先级最高的**一个问题**。
2. 用 AskQuestion 工具呈现 2–4 个互斥选项；把建议项放在第一位并标注“推荐”。允许用户通过 Other 自由回答。
3. 为每个选项说明会怎样改变执行做法。不要把多个问题塞进一个选项。
4. 提问后立即停止，等待用户回答；不得在同一轮继续提问、产出方案或开始实现。
5. 收到答案后，记录：
   - 已确认答案
   - 被消除的已知未知
   - 对范围、方案或验收标准的具体影响
6. 根据新答案删除失效问题、加入新暴露且会改变执行的问题，再进入下一轮。

## 收敛条件

满足以下条件后停止访谈：

- 不再存在会改变执行做法的未回答问题
- 范围、关键约束、失败策略和验收标准均已确定
- 剩余不确定性都有明确默认值，且默认值不会改变总体方向

不要为了“更完整”继续询问不影响执行的问题。

## 最终产出

访谈结束后输出：

```markdown
## 访谈结论：<需求一句话>

### 已确定的已知未知
- <问题> → <答案> → <对执行的影响>

### 最终需求
- 必须实现：
- 明确不做：
- 关键约束：
- 失败与回滚策略：

### 验收标准
- <可验证条件>

### 采用的默认值
- <未继续提问的不确定项>：<默认值及理由>

### 执行就绪
<是否可进入 reference；若否，说明唯一阻塞项>
```

若“执行就绪”为是，携带这份结论进入 reference，核对文档、代码和测试并建立“已知的已知”基线；不要直接进入 planning 或实现。
