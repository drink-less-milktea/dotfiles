# 深度增强论文笔记检查清单

用于把已生成的论文 HTML note 从“中文化摘要”升级为“可用于汇报/综述的深入讲解”。

## 目标章节

只需要重点增强：

- `核心技术原理讲解`
- `实验分析`

保留原 HTML/CSS、元信息表、图片引用、PDF、`paper.txt` 和 `assets/` 不变，除非用户明确要求重建。

## 核心技术原理讲解

每篇至少覆盖以下要点：

1. **设计动机**：传统方法为什么不够好，瓶颈是显存、带宽、延迟、检索漂移、调度空转还是质量下降。
2. **方法机制**：按“输入/状态 → 计算步骤 → 输出/缓存/调度决策”讲清数据流。
3. **公式或表达式**：优先从论文中提取；若论文没有显式公式，可以给出可由论文定义支撑的缓存量、复杂度或保留率表达，例如 `(1-r)n`、`O(n)`、`KV size = layers × heads × dim × tokens`。
4. **与传统方法区别**：明确对比 MHA/GQA/MQA、full KV、token-level pruning、FCFS scheduling、CPU-side retrieval 等 baseline 类别。
5. **收益与代价**：说明提升来自哪里，以及可能牺牲的质量、额外 kernel、元数据、训练成本或调度复杂度。

## 实验分析

每篇至少覆盖以下要点：

1. **实验设置**：模型、数据集、任务、硬件或推理框架，缺失时写“论文中未明确标注”。
2. **baseline**：列出主要比较对象，不要只写“传统方法”。
3. **关键数字**：必须包含论文中可核对的具体数值，如吞吐、延迟、准确率、困惑度、压缩率、显存占用、TTFT。
4. **相对提升**：优先写论文给出的提升；如果基于表格可直接计算，说明“由表格数值计算”。不要凭印象编造百分比。
5. **消融结论**：说明哪个模块/超参带来收益，哪些场景收益更大或更小。
6. **边界条件**：图中才有、正文未逐项列出的数字，不要臆测；写“需回到 PDF 图表核对”。

## 验证要求

增强后至少检查：

- 两个目标章节都存在。
- 每个目标章节有足够内容，通常不少于 6 条要点或等价段落。
- 实验分析包含多个数字，不只是定性结论。
- 技术原理包含公式/复杂度/缓存量/保留率等可核对表达。
- 图片引用仍存在且路径不变。
- 没有把英文 PDF 长段落复制进正文；英文仅保留在论文标题、方法名、模型名、数据集名、指标名、公式和 URL 中。

## 常见坑

- 把“英文原文证据摘录”留在 HTML 里：必须改成中文转述 + 页码/图表编号。
- 把 baseline 写得过泛：应尽量写出具体系统或方法名。
- 为了显得详尽而编造数字：宁可写“论文中未明确标注”。
- 只增强摘要，不增强技术原理和实验分析：用户明确要求这两节最详尽。
