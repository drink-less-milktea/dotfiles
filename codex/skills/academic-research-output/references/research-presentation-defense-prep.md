# Research Presentation Defense Supplement Pass

Use this when a user is close to a research presentation or first group report and asks what to add for possible questions, or asks to update an outline/PPT description for defense readiness.

## Workflow

1. Inspect the current outline, slide descriptions, reading list, and note index before suggesting additions.
2. Identify gaps by likely committee/advisor questions, not by adding more papers indiscriminately:
   - foundational formulas and back-of-the-envelope estimates;
   - boundary distinctions between commonly confused methods;
   - evaluation protocol and fairness questions;
   - known limitations and failure cases;
   - future-work positioning.
3. Keep the main narrative stable when the presentation is imminent. Add material as backup slides, speaker notes, Q&A sections, or reading-list additions unless the user explicitly wants a full restructure.
4. When the user asks a specific technical question (e.g., "what does X mean in this context?"), do not give a generic definition. Read the primary source (paper text, not just abstract) to extract the exact rationale, then answer with source-backed wording and a ready-to-use defense block. This prevents mischaracterizing domain-specific terminology.
5. When the user asks a yes/no question about a method's effect (e.g., "does X significantly help Y?"), avoid a simple yes/no. Give a conditional answer: "helpful under condition A, less so under condition B," with the mechanism explained. This matches how committees probe for depth of understanding.
4. Add only a small number of high-leverage supplementary papers. Prefer papers that support answerability: foundational background, baseline/benchmark, or conceptual boundary papers.
5. For each added paper, create a note that includes a short "答辩可用问答" section: likely question, answer hook, and where it fits in the talk.
6. Update the paper index/manifest and verify that added PDFs, notes, and outline references exist.

## Good Supplement Categories

- Formula/scale estimate page: lets the user answer "how big is this problem really?"
- Boundary page: distinguishes adjacent methods that may be confused.
- Benchmark page: explains how claims should be evaluated fairly.
- Misconception page: records what not to overclaim.
- Future-work page: turns limitations into credible next steps.
- Method-detail Q&A page: when the user asks a specific technical question (e.g., "what does asymmetric quantization mean here?"), produce a concise answer with primary-source evidence and a ready-to-use defense wording block. Add the question/answer pair to the Q&A section of the relevant artifact so it is not lost.

## Pitfalls

- Do not bloat the main 45-minute narrative right before the presentation; use backup material.
- Do not add papers only because they are recent; add papers that answer likely questions.
- Do not present benchmark success as universal quality; spell out task/workload coverage and gaps.
- Do not leave new notes disconnected from the outline or PPT explanation files.
- When the user asks a specific technical question (e.g., "what does X mean in this context?"), do not give a generic definition. This prevents mischaracterizing domain-specific terminology.
- When the user asks a yes/no question about a method's effect (e.g., "does X significantly help Y?"), avoid a simple yes/no. Give a conditional answer: "helpful under condition A, less so under condition B," with the mechanism explained. This matches how committees probe for depth of understanding.
