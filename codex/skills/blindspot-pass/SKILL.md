# Blindspot Pass（盲区扫描）

在动手实现之前，用多个并行子代理从正交视角扫描一个模糊需求，暴露出提问者和回答者都没意识到的风险与约束。

## 何时触发

满足任一条件才启动本流程，否则直接正常做任务：

- 需求可以有多种差异巨大的合理解读（改哪里、改多大、为谁改都不明确）
- 任务会波及多个模块/系统，但用户只给了一句话描述
- 用户明确要求盲区扫描或"先调研再动手"

**不要**对简单明确的小任务启动本流程（如改一行配置、修一个指明位置的 bug）。

## 工作流

### 第 1 步：解构需求

先用一两句话写下你对需求的默认解读，然后列出该解读依赖的隐含假设。这些假设就是扫描目标。

### 第 2 步：并行派出调研子代理

从下表选 3–6 个与任务相关的维度，**在同一批次**用 Task 工具并行启动子代理（`explore` 用于代码库内调研，`generalPurpose` 用于需要搜索外部信息的维度），全部 `readonly: true`：

| 维度 | 调研问题 |
|---|---|
| 代码库现状 | 相关模块的真实结构、已有类似实现、命名与惯例 |
| 隐藏耦合 | 谁依赖这块代码？改动会波及哪些调用方、配置、数据格式？ |
| 边界与异常 | 并发、空值、超大输入、失败路径、回滚，现有代码怎么处理？ |
| 历史与意图 | git 历史、注释、文档中这块代码为什么长这样？有没有踩坑记录？ |
| 领域惯例 | 业界对这类问题的标准做法、已知陷阱（需要 web 搜索时用 generalPurpose） |
| 替代解读 | 用户这句话还能怎么理解？哪种解读实现代价差异最大？ |

子代理 prompt 模板（每个代理只给一个维度，避免结论互相污染）：

```
背景：用户提出需求「<原话>」，我的默认解读是「<解读>」。
你的任务：只从「<维度>」这一个角度调研 <代码库路径/主题>。
不要给实现方案。返回：
1. 与该维度相关的客观事实（带文件路径或来源）
2. 会推翻或收窄默认解读的发现
3. 该维度下用户可能没意识到的风险，按严重程度排序
```

### 第 3 步：汇总盲区报告

等全部子代理返回后，去重、交叉验证冲突结论，按此结构输出：

```markdown
## 盲区报告：<需求一句话>

### 默认解读
<你准备按什么理解去做>

### 已确认事实
- <子代理带证据的发现，附文件路径>

### 已知未知（我们知道自己不知道）
- <需要用户或进一步调研才能回答的问题>

### 未知的未知（扫描新暴露的盲区）
- 🔴 <会改变方案方向的发现>
- 🟡 <会影响实现细节的发现>
- 🟢 <值得留意但不阻塞的发现>

### 澄清问题（按决策影响排序，最多 3 个）
1. <只问会实质改变做法的问题>
```

### 第 4 步：收敛

- 若 🔴 项为空且澄清问题都有合理默认值：陈述默认选择，将盲区报告交给 brainstorm
- 若存在 🔴 项或解读分歧巨大：呈现报告，等用户回答关键问题后再交给 brainstorm

## 注意事项

- 子代理之间维度必须正交，不要派两个代理查同一件事
- 子代理一律只读，扫描阶段不做任何修改
- 报告里每条"事实"必须带出处（文件路径、commit、URL），无出处的写入"已知未知"
- 整个扫描以一轮为限，不要递归派代理
