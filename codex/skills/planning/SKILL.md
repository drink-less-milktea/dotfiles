# Planning

把已澄清的需求转成可确认的实施计划。先锁定“选错最贵”的决定，再展开稳定细节。

## 进入 Plan Mode

1. 立即调用 `SwitchMode` 切换到 `plan`；已经处于 Plan Mode 时不要重复调用。
2. 计划获批前仅做只读调研，不编辑文件、不提交、不执行外部写操作。
3. 确认 reference 已完成，并按流程顺序汇总输入：
   - blindspot-pass 的盲区报告
   - brainstorm 的需求确认单
   - interview 的访谈结论
   - reference 的参考基线
4. 缺少 reference 参考基线时先返回 reference；其他输入缺失仅在会改变高代价决策时返回对应流程。

## 建立决策清单

从需求和代码库中提取真正需要决定的事项。为每项判断：

- **返工代价**：选错后需要重写多少代码、迁移多少数据、协调多少系统
- **推翻概率**：需求、技术可行性或外部依赖使其返回重做的可能性
- **影响半径**：影响多少模块、接口、用户或团队
- **可逆性**：能否低成本切换、回滚或并行保留
- **阻塞性**：是否决定其他步骤的形态

优先级遵循：

`高返工代价 + 高推翻概率 + 大影响半径 + 低可逆性 + 强阻塞性`

不要为了制造“决策”而罗列已经由事实或用户确认固定的事项。

## 先处理高代价决策

把优先级最高的决策写在计划最前面。每项必须包含：

```markdown
### 决策 N：<要决定什么>
- 推荐：<唯一明确选择>
- 理由：<基于哪些事实、约束和权衡>
- 选错代价：<会返工或破坏什么>
- 验证方式：<如何在全面实现前低成本证伪>
- 回滚边界：<何时还能安全改变决定>
```

若存在多个会显著改变计划的有效选择，不要把选项留在计划中。先向用户提出 1–2 个关键问题，得到答案后再制定具体计划。

## 再安排执行步骤

决策展示顺序与执行顺序分开处理：

- **决策展示顺序**按风险和返工代价排序
- **执行步骤顺序**按依赖关系排序，并尽早安排可验证高风险假设的 spike、测试或原型

每个执行步骤写清：

1. 修改的具体文件或模块
2. 要改变的行为、接口或数据流
3. 依赖的前置决策和步骤
4. 需要新增或更新的测试
5. 完成与验证标准
6. 必要的迁移、兼容或回滚处理

计划必须具体到另一位工程师无需重新做架构决策即可执行。

## 最后写固定决定

把下列内容放在计划后部的“固定决定与约束”：

- 已被用户明确确认的要求
- 由现有契约、平台或规范强制决定的内容
- 不会改变总体方案的命名、格式和机械性细节
- 已采用且无需再次讨论的默认值

固定项仍需附来源，但不要让它们淹没高代价决策。

## 输出格式

```markdown
# <计划标题>

## 高代价决策（按返工风险排序）
### 决策 1：...

## 实施步骤（按依赖顺序）
1. <文件/模块>：<具体变更与验证>

## 固定决定与约束
- <决定>（来源：<参考基线/访谈/规范>）

## 验证与回滚
- 自动化测试：
- 集成验证：
- 回滚触发条件与方法：
```

不要包含 TBD、“可选”“视情况而定”或多个未选择方案。计划完成后使用当前环境的计划确认机制提交给用户批准；只有批准后才进入实现。
